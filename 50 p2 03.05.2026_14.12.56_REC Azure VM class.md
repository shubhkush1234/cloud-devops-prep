## Detailed Notes & Architectural Walkthrough

### 1. Storage provisioning & Resource Pricing

1. When initiating the creation of a virtual machine in the Azure Management Portal, the storage subsystem default profile assigns a standard image storage disk (typically 30 GB).
2. Modifying the OS storage type to high-capacity options, such as a 2 TiB Premium SSD, alters the **Estimated monthly cost** display panel dynamically, shifting estimated rates upward significantly (e.g., from ~$86.31 to over ~$340.10 per month).
3. Under the storage attachment parameters, additional standalone storage can be attached via the **Create and attach a new disk** dialog.
4. Specifying an additional 1024 GiB Premium SSD data volume forces an immediate adjustment in the resource pricing calculator matrix, adding separate monthly storage overhead penalties to the subscription billing profile.
5. To maintain cost-efficiency during non-production training cycles, any heavy data disks provisioned accidentally should be unlinked or dropped entirely prior to validation to avoid exhausting credit limits prematurely.

### 2. Virtual Network & Subnet Segmentation Architecture

1. Transitioning to the **Networking** parameters tab requires mapping the new virtual asset to a primary infrastructure router, known as a **Virtual Network (VNet)** (e.g., matching the pre-existing network asset named `dlink`).
2. If the chosen VNet contains no predefined internal partitions, the interface displays an empty state; allocation is completed by selecting the **Manage subnet configuration** link.
3. This action opens the standalone VNet configuration blade, where selecting **+ Subnet** introduces a brand-new localized broadcast domain (e.g., allocating an entity named `netflix-subnet`).
4. The system demands a precise network slice specified in CIDR notation (e.g., applying the block range `10.143.0.0/24`), which effectively builds an infrastructure slice capable of accommodating up to 251 dynamically assignable host addresses.
5. Saving this component registers the network partition dynamically, allowing you to return to the core VM deployment wizard and seamlessly map the new Virtual Machine directly into the freshly updated `netflix-subnet` boundary.

### 3. Traffic Isolation Logic (Private vs. Public Space)

```
[Internet Source: Local Client Machine] ---> (Public IP: 4.204.224.131) via WAN
                                                          │
                                                    [Azure VNet Boundary]
                                                          │
                                        (Private IP: 10.143.0.4) via LAN Subnet
```

1. Upon successful generation of the node `netflix-vm` inside the designated target group `rg-dev`, Azure assigns a distinct dual-IP signature profile to the virtual network adapter card.
2. The **Private IP Address** (e.g., `10.143.0.4`) is automatically carved directly out of the internal `netflix-subnet` block range and remains strictly un-routable over the open web.
3. This Private IP acts as the primary secure mechanism for uniform server-to-server data distribution across internal production tiers within the shared VNet landscape.
4. The **Public IP Address** (e.g., `4.204.224.131`) serves as the dedicated public destination entry point mapped across the external WAN internet space.
5. External devices, such as an administrator working from a local home network environment routed behind an external ISP gateway (e.g., subnet `192.168.0.0/24`), must target this public WAN address directly to tunnel traffic into the isolated cloud platform.

1. Remote command-line terminal interactions targeting a **Linux-based host system** utilize the encrypted **Secure Shell (SSH)** communication framework.
2. The administrator triggers a connection sequence by formatting the execution script inside a local terminal environment exactly as follows: `ssh username@public_ip_address` (e.g., `ssh devopsadmin@4.204.224.131`).

### 5. Web Platform Deployment & Firewall Rule Orchestration

```
Incoming Client Request (Port 80) ──> [ NSG Firewall Layer ] ──X (Dropped by Default Rule)
                                                │
                          [ Apply Rule: Allow Priority 100 (Port 80) ]
                                                │
                                                └───> [ Targets Web Server Node: Nginx OK ]

```

1. Once authenticated safely into the underlying Linux instance via an open shell, executing `sudo apt update` synchronizes system directory links with the primary distribution download mirrors.
2. Installing an open-source web server application layer is achieved by executing the system orchestration directive `sudo apt install nginx -y`.
3. If an administrator attempts to paste the raw destination Public IP address directly into a standard consumer web browser tab immediately after installation, the browser canvas stalls and returns an `ERR_CONNECTION_TIMED_OUT` diagnostic notification.
4. This systematic block occurs because the Azure engine automatically shields every new public node behind an immutable default **Network Security Group (NSG)** filter boundary that drops all uninvited inbound WAN traffic.
5. To bridge this traffic gap, navigate to the target virtual machine's settings panel in the Azure Management Portal and choose the **Network settings** control menu.
6. Select the **Add inbound port rule** configuration control matrix to create an explicit routing path.
7. Set the **Destination port ranges** criteria to explicitly match target value `80`, toggle the target platform protocol transport value to **TCP**, switch the execution action value to **Allow**, and click the confirmation button.
8. The newly injected baseline permission rule takes absolute execution precedence over the catch-all `DenyAllInBound` rule, immediately prompting the browser application to clear the timeout error and successfully render the default Nginx landing interface.

## Interview Ready Answers Section

### 1. NSG Rule Processing & Conflict Resolution (Amazon)

1. Azure Network Security Groups resolve policy matches based on strict, incremental numerical order using priority ranking indexes that scale from 100 up to 4096.
2. The security layer processes traffic sequentially; the rule with the absolute lowest assigned integer value is evaluated first.
3. Once a traffic condition matches all defined search parameters (matching source IP, target port, and transport protocol), processing terminates immediately, and the rule's action is applied.

```
       [ Evaluates Priority: 100 ] ───( No Structural Parameter Match )───┐
                                                                          ▼
       [ Evaluates Priority: 200 ] ───( Target Parameter Matched! )───> [ Enforces Action & Halts Evaluation ]
                                                                          ▼
       [ Evaluates Priority: 300 ] ───( Skipped entirely - Never Read ) ──┘

```

### 2. Stateful Filter Mechanics vs. Stateless Controls (Microsoft)

1. Azure NSG structures operate using stateful evaluation mechanics.
2. When an inbound rule permits passage to a specific service port, the firewall platform notes the connection state in its internal session tracking directory.
3. Outbound responses bound to that specific session pass through the perimeter automatically, completely bypassing any outbound restrictions.
4. Conversely, stateless firewalls lack memory architecture, which requires administrators to write mirror-image inverse permissions for both inbound and outbound paths to ensure a single working connection.

```
Inbound Packet Allowed ──> [ Stateful Memory Directory Update ] ──> Server Process Completed
                                                                             │
Outbound Reply Allowed <── [ Matches Logged Directory Session ] <────────────┘

```

### 5. Multi-Layer Security Architecture (Infosys)

1. When traffic targets a virtual resource that is bound by both a parent subnet protection rule and a localized NIC firewall policy, Azure evaluates both security perimeters sequentially.
2. Inbound network packages hit the parent subnet security boundary first.
3. If the subnet-level policy permits entry, the traffic passes down to the internal link layer, where it encounters the localized NIC security filter rule set.
4. Outbound network traffic follows the exact inverse path, hitting the localized NIC rule definitions first before passing out through the parent subnet filter layout.

```
Inbound Traffic ──> [ Subnet-Tier NSG Rules ] ──( Allowed )──> [ Local NIC-Tier NSG Rules ] ──> Compute Target
                                                                                                      │

```

### 7. Non-Secure HTTP Transport Routing Framework (Accenture)

1. Standard web browsing clients connect over Port 80 by default for standard unencrypted web traffic communications.
2. Azure blocks this traffic by default using a baseline rule named `DenyAllInBound` (indexed at priority 65500).
3. To open a public web path, an administrator adds a new permission rule configured with a lower priority index value (e.g., priority 100).
4. Because the security group processes rules in order from lowest to highest priority index, the new permit rule triggers a match first, allowing the web request to reach the server.

```
Incoming Web Request ──> [ Priority Rule: 100 (Allow Port 80) ] ───> ( Match Found! Traffic Passed )
                                                                               │
                          [ Default Rule: 65500 (Deny All Inbound) ] <──( Evaluation Ceased )

```

## Architectural Enhancements & Best Practices

1. **Enterprise Identity Federation (Microsoft Entra ID Integration):** Move away from using static administrative passwords on individual server images. Instead, connect your systems directly to central identity platforms like Microsoft Entra ID. This allows you to manage system access using secure, short-lived tokens and multi-factor authentication (MFA).
2. **Just-In-Time (JIT) Port Locking Strategy:** Keep management ports like SSH (22) and RDP (3389) locked down by default within your security groups. Use Just-In-Time (JIT) access policies to temporarily open these paths only when a verified administrator requests access, automatically closing the port after a set window.
3. **Advanced Web Application Firewalls (WAF):** Do not expose your frontend applications directly to the open internet using basic public routing. Instead, route your incoming traffic through an intermediate layer like an Azure Application Gateway equipped with a Web Application Firewall (WAF) to inspect and block common web application vulnerabilities (such as SQL injection or cross-site scripting) before they can reach your servers.
