## 1. To-the-Point Summary

In this session, the focus was entirely on setting up an **end-to-end architecture** on Microsoft Azure to implement high availability and outbound connectivity. The process involved creating a Virtual Network (**VNet**) with dedicated subnets for front-end processing and secure management. Two backend Linux virtual machines were deployed inside a private subnet (with no direct public IPs) to securely host **Nginx web servers**.

To management-access these instances safely, an **Azure Bastion** service was set up. Additionally, an **Azure NAT Gateway** was introduced on the subnet level to establish crucial outbound internet connectivity for configuration updates. Finally, an **Azure Public Load Balancer** was deployed to expose the applications securely to the internet, configured to route traffic smoothly across backend resources using a round-robin algorithm.

## 2. Step-by-Step Infrastructure Deployment Notes

The session systematically outlined the design and configuration of an architecture designed around real-world production environments.

### Phase 1: Network Foundation (VNet & Subnet Sizing)

* **Virtual Network Creation**: Define a central space with the address block `10.124.0.0/16`.
* **Subnet Partitioning**: To maintain isolation and separate distinct tiers of service, two discrete subnet ranges are designated:
* **Frontend Subnet**: Configured with a `10.124.0.0/24` range, hosting multi-tier internal applications.

### Phase 2: Compute Deployment & Secure Management Entry

* **Internal VM Provisioning**: Set up two Linux VMs (`frontend-vm-01` and `frontend-vm-02`) inside the Frontend Subnet.
* **Public IP Constraint**: Both backend virtual machines are created with their Public IP explicitly set to **None**. This forces an absolute requirement for private network traversal.
* **Bastion Host Connection**: An Azure Bastion resource is configured on the `AzureBastionSubnet`. This configuration allows administrators to safely access the instances using the Azure Portal directly through secure web-based browser tabs over SSH on Port 22, rendering external jump boxes obsolete.

### Phase 3: Provisioning Outbound Connectivity (NAT Gateway)

* **Problem Statement**: When virtual machines are completely locked in private environments with no public endpoints, executing a package check (e.g., `sudo apt update`) fails or times out.
* **Solution Implementation**: Create an Azure NAT Gateway (`nat-dhondhu`) tied to a newly provisioned Public IP.
* **Subnet Mapping**: Attach the NAT Gateway directly onto the **Frontend Subnet**. Once associated, any instance inside that subnet can automatically fetch external mirrors and pull source binaries without revealing open incoming connections.

### Phase 4: Constructing the Load Balancer Architecture

To successfully run and expose the network workloads, the Azure Load Balancer was set up through 4 core structural pillars:

1. **Frontend IP Configuration**: The entry gateway that maps to a static Azure Public IP. It provides the initial point of contact for clients trying to access the cluster.
2. **Backend Pool**: A designated grouping of target instances. The network interfaces (NICs) of both `frontend-vm-01` and `frontend-vm-02` are placed inside this pool.
3. **Health Probes**: Periodic network heartbeat checks monitored on TCP Port 80 to establish target reachability. If a node stops responding, traffic stops routing to it.
4. **Load Balancing Rules**: The deterministic matching logic that combines incoming port rules with a backend target, processing web queries using a balanced **Round Robin** algorithm.

### Q1. How does Azure Load Balancer handle session persistence, and what options are available to adjust this behavior? (Asked at: Microsoft, Accenture) `[Important]`

#### Interview-Ready Answer

Azure Load Balancer by default uses a **5-tuple hash** algorithm composed of Source IP, Source Port, Destination IP, Destination Port, and Protocol Type to distribute client connections. Because of this default behavior, a subsequent request from the same user could land on a completely different backend instance if the source port randomizes.

To overcome stateless distribution, Azure provides two stateful session persistence settings:

* **2-Tuple (Client IP)**: Directs future requests originating from the same source IP to the exact same backend instance, irrespective of port transformations.
* **3-Tuple (Client IP, Protocol)**: Ensures requests from the same client using the same protocol type remain pinned to a specific worker machine.

```
          │
          ▼
┌───────────────────┐
│   Load Balancer   │ ───► Check Hash (Client IP Persistence)
└───────────────────┘
     │         │
     ▼         ▼
┌─────────┐ ┌─────────┐
│  VM 01  │ │  VM 02  │ (Pinned sticky host determined by Client IP)
└─────────┘ └─────────┘

```

### Q2. Describe the processing mechanics of an Azure NAT Gateway versus Azure Load Balancer outbound rules. When should you choose one over the other? (Asked at: Wipro, Capgemini)

#### Interview-Ready Answer

While both resources provide secure SNAT translation pathways for private subnets, their behaviors differ significantly:

* **Azure NAT Gateway**: This is a fully managed network-centric resource designed explicitly for highly scalable **outbound-initiated** flows. It supports up to 64,000 concurrent sessions per mapped public IP across an entire subnet layer, drastically eliminating SNAT port exhaustion.
* **Load Balancer Outbound Rules**: These tie outbound translations directly to specified backend pools rather than the underlying subnet layout. They are useful for architectures where nodes require symmetrical routing (sending traffic out through the exact same IP address that manages incoming web requests).

> **Architectural Recommendation**: For enterprise workloads with a high volume of external API calls, choose a NAT Gateway to maximize connection resilience.

### Q3. What is the fundamental difference between Azure Standard Load Balancer and Basic Load Balancer regarding Security Group bindings? (Asked at: Infosys, HCL) `[Important]`

#### Interview-Ready Answer

The difference centers around the security baseline design of the SKUs:

* **Basic Load Balancer**: Open by default. Mapped virtual networks do not mandate an explicit Network Security Group (NSG) configuration to accept frontend connection tracking.
* **Standard Load Balancer**: **Closed by default**. The architecture operates on zero-trust principles. All data paths hitting backend pools are completely blocked unless an explicit NSG rule is attached to allow inbound communication. Standard SKUs also support advanced capabilities like HTTPS health probes, Availability Zones, and HA ports.

```
Basic LB:    [Public Traffic] ──► [Basic LB] ──► [VM Pool (Open by Default)]

Standard LB: [Public Traffic] ──► [Standard LB] ──► [NSG Firewall (Required Block/Allow Check)] ──► [VM Pool]

```

### Q4. How do you troubleshoot a scenario where an Azure Load Balancer shows all backend instances as 'Unhealthy' via Health Probes? (Asked at: Tata Consultancy Services, Cognizant) `[Important]`

#### Interview-Ready Answer

When an entire pool registers an unhealthy status, I execute a structured layer-by-layer diagnostic workflow:

1. **Host-Level Service Verification**: Access the target instances via Azure Bastion and check if the process is running using `sudo systemctl status nginx`.
2. **Local Listening Verification**: Confirm that the application is binding to the correct port locally by executing `netstat -tuln | grep 80`.
3. **Network Security Group Analysis**: Check Azure NSG rules to ensure probe traffic originating from the core Azure infrastructure IP block (`168.63.129.16`) is not accidentally blocked.
4. **Local Firewall Inspection**: Audit local operating system filtering tables (such as `ufw` or `iptables`) to ensure they allow network probing paths.

### Q5. Explain the architecture of an Azure Bastion host. How does it isolate private infrastructure from the public internet? (Asked at: Cloud consultations, DXC Technology)

#### Interview-Ready Answer

Azure Bastion is a fully managed PaaS element deployed inside a dedicated subnet named `AzureBastionSubnet`. It functions as an isolated RDP/SSH proxy layer inside the Azure edge network perimeter.

Users authenticate securely against the Azure portal using a web browser over HTTPS (Port 443). Once authorized, the Bastion host handles the target connection over private IP lines using internal backend interfaces. This eliminates the need to expose virtual machines directly to the public internet via raw public SSH ports, significantly reducing the environment's attack surface.

### Q6. What are Azure Application Gateway 'HA Ports' alternatives, and how do you achieve multi-port load balancing in Layer 4 Azure Load Balancers? (Asked at: IBM, Oracle)

#### Interview-Ready Answer

Azure Layer 4 Standard Load Balancers support an option called **HA Ports** (High Availability Ports) exclusively when configured as an internal network appliance cluster. This allows a single frontend configuration to scale out and process all flows across every TCP and UDP port concurrently.

For public-facing Layer 4 architectures where HA ports are not supported, engineers must configure discrete **Load Balancing Rules** for each application port or switch to an **Azure Application Gateway** (Layer 7 alternative) to handle multi-port web proxying safely.

### Q7. If an Azure Application is experiencing SNAT Port Exhaustion, how can this be diagnosed and mitigated? (Asked at: Amazon Web Services, Tech Mahindra) `[Important]`

#### Interview-Ready Answer

SNAT port exhaustion occurs when backend nodes initiate a large volume of outbound requests to external endpoints, consuming all pre-allocated source ports. This causes outbound connections to drop or fail.

* **Diagnosis**: Monitor the `SNAT Connection Count` and `Allocated SNAT Ports` metrics inside Azure Monitor.
* **Mitigation**: The most effective mitigation strategy is to attach a managed **Azure NAT Gateway** to the workload's subnet. A NAT Gateway dynamic allocation approach leverages up to 64,000 ports per public IP address, preventing port exhaustion. Alternatively, you can explicitly scale out the environment by assigning multiple public IPs to the load balancer's outbound pool.

```
[ Scaled Outbound Requests ] ──► [ Static SNAT Pool Limit ]  ──► Exhaustion Drop!
[ Scaled Outbound Requests ] ──► [ Azure NAT Gateway Pool ] ──► Dynamic Scale OK!

```

### Q9. How do you upgrade an Azure Basic Load Balancer to a Standard Load Balancer with zero downtime? (Asked at: PwC, Cisco)

#### Interview-Ready Answer

Upgrading an active Basic Load Balancer directly to a Standard SKU is not supported due to differences in their underlying platform architectures. To perform this migration with zero down-time, I follow a side-by-side transition strategy:

1. **Parallel Deployment**: Deploy a completely new Standard Load Balancer alongside the active Basic instance, using a new public IP.
2. **Synchronize Configurations**: Replicate all backend pools, health probes, and load-balancing rules on the new Standard instance.
3. **Apply NSG Controls**: Apply appropriate Network Security Group (NSG) permissions to allow data paths to flow to the backend instances on the Standard instance.
4. **DNS Failover**: Update upstream corporate DNS definitions or Traffic Manager pointer records to route client traffic to the new Standard entry point, slowly decommissioning the old Basic instance once traffic has fully migrated.

```
Old Path: [ Users ] ──► [ Existing DNS ] ──► [ Basic LB ] ──► [ Virtual Machines ]
                                                                      ▲
New Path: [ Users ] ──► [ Updated DNS ]  ──► [ Standard LB ] ─────────┘

```

### Q10. What is an Floating IP (Direct Server Return) inside Azure Load Balancer, and when is it required? (Asked at: VM Ware, NTT Data)

#### Interview-Ready Answer

Floating IP, also known as **Direct Server Return (DSR)**, changes how the return data path is routed in load-balanced workloads.

* **Disabled (Default)**: Symmetrical network routing is used. The target instance rewrites the response address, returning traffic to the load balancer frontend node first before it heads back to the client.
* **Enabled (DSR)**: Symmetrical routing is bypassed. The virtual machine shares the exact same IP configuration as the load balancer's frontend. The node responds directly to the client over external networks, completely bypassing the load balancer processing engine.

> **Use Case**: This is commonly used when hosting SQL Server AlwaysOn Availability Groups or high-volume multimedia streaming services to avoid bottlenecks at the load balancer layer.

```
Inbound Path:  [ Client ] ──────► [ Load Balancer ] ──────► [ Virtual Machine Cluster ]
                                                                  │
Outbound Path: [ Client ] ◄───────────────────────────────────────┘ (Direct Server Return)

```

## 5. Architectural Architect Tips & Intelligent Suggestions

As a DevOps Architect with over two decades of industry experience, here are three high-level best practices for managing enterprise cloud infrastructure:

* **Enforce Zero-Trust Network Topologies via Standard SKUs**: Never skimp on infrastructure budgets by selecting Basic network infrastructure blocks. Standard load balancers provide default-deny security baselines that protect backend pools from zero-day external scans. Always combine these elements with a managed NAT Gateway on the subnet layer from day one to safeguard outward-initiated tracking routines against SNAT port exhaustion.
* **Automate Infrastructure via Declarative Code**: Avoid manually clicking through the Azure Portal during live deployments. Doing so introduces configuration drift and makes environments difficult to replicate. Instead, build your infrastructure using declarative tools like **Terraform** or **OpenTofu**. This allows you to manage network layouts, security group rules, and load balancer settings as versions within your CI/CD pipelines.
* **Decouple Storage Layers for High Availability**: Stateful architectures can lead to configuration silos when servers go down. Ensure your local instances remain stateless by shifting processing tasks to shared storage options like **Azure NetApp Files** or premium **managed database instances**. This ensures that if a component fails, replacement instances can be spin up instantly via automated scaling actions without risking data loss.
