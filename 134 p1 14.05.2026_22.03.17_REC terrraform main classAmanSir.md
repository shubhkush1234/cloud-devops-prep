# Comprehensive Study Guide: Modern DevOps, AI-Ops & Azure Terraform

## To-the-Point Summary

This lecture serves as a strategic and technical roadmap for transitioning from legacy infrastructure to modern, AI-driven DevOps practices. The trainer emphasizes the necessity of mastering fundamental Cloud (Azure) and Infrastructure as Code (Terraform) skills before attempting to deploy advanced architectures. The session visually maps the evolution of applications—from Monolithic (Legacy/Virtualization) to Microservices (Containerization/Serverless)—and ultimately introduces the integration of Machine Learning and Agentic AI models into these environments. Practical technical steps focus on provisioning a 3-tier architecture in Azure using progressively advanced Terraform variable structures.

## Detailed Lecture Notes & Key Technical Steps

### 1. The Mindset and Prerequisites for Modern DevOps

* **The "Patience" Principle:** Moving into AI-Ops and ML-Ops requires a solid foundation in core DevOps. Skipping fundamental steps (like manual portal creation or basic scripting) leads to failures in advanced automation.
* **Embracing Automation:** Do not rely solely on UI click-ops. All infrastructure must eventually be managed via code (Terraform) and deployed through pipelines.
* **The Paradigm Shift:** The industry is moving beyond standard CI/CD. DevOps engineers must now learn how to host and integrate AI models (Agentic AI, LLMs, Machine Learning) within containerized and serverless environments.

### 2. Architectural Evolution Mapping

The trainer maps out the transition of application hosting:

* **Monolithic Applications (Legacy):** Traditionally hosted on bare metal, moving to **Virtualization** (VMs), and eventually retrofitted into **Serverless** wrappers.
* **Microservice Applications (Modernization):** Highly decoupled architectures moving naturally into **Containerization** (Docker/Kubernetes) and natively designed for **Serverless** execution.
* **AI Integration:** Under the microservices model, dedicated services are spun up for **Machine Learning** predictions and **Agentic AI** servers (e.g., using LangChain or MCP servers).

*[Screenshot: Flowchart detailing the split between Monolithic (Legacy) and Microservice (Modernization) applications, branching into Virtualization, Containerization, Serverless, ML, and Agentic AI.]*

### 3. Key Technical Steps: Azure Infrastructure Setup

To build a functional 3-Tier Architecture in Azure, the following hierarchy must be provisioned:

1. **Create a Resource Group (RG):** The logical container for all assets.
2. **Create a Virtual Network (VNet):** Define the private network space within the Azure region.
3. **Create a Subnet:** Partition the VNet (e.g., a public subnet for front-facing assets).
4. **Provision a Virtual Machine (VM):** Deploy a `frontend-vm` within the subnet and install web services (e.g., Nginx) to handle incoming traffic.

*[Screenshot: Azure Infrastructure tree showing Resource Group -> Virtual Network -> Public Subnet -> frontend-vm.]*

### 4. Key Technical Steps: Terraform Variable Mastery

The trainer outlines a progressive roadmap for writing Terraform code, moving from basic to highly scalable dynamic configurations:

1. **Input Hardcoded:** Writing static values directly into the `.tf` resource blocks (not recommended for production).
2. **Variable as a String:** Passing single dynamic values.
3. **Variable as a List(String):** Passing ordered lists of values for bulk resource creation.
4. **Variable as a Map:** Utilizing key-value pairs for structured data.
5. **Variable as a `for_each` (Map):** Iterating over a map to dynamically create multiple unique resources without duplicating code blocks.
6. **Variable as a `for_each` (Nested-Map):** The most advanced pattern, used for deploying complex, multi-layered infrastructure components simultaneously.

## Interview Questions & Answers

**1. What is Terraform, and how does its architecture work?**

* **Company:** Amazon, TCS
* **Answer:** Terraform is an open-source Infrastructure as Code (IaC) tool by HashiCorp that allows users to define and provision data center infrastructure declaratively. Its architecture consists of two main parts: **Terraform Core**, which parses configurations and manages state/execution plans, and **Terraform Providers**, which are plugins that interface directly with cloud APIs (like Azure, AWS, GCP) to manage the underlying resources.

**2. Explain the difference between Monolithic and Microservices architecture.**

* **Company:** Accenture, Cognizant
* **Answer:** A Monolithic architecture builds all application components (UI, business logic, data access) into a single, tightly coupled codebase and deployment artifact. Microservices architecture breaks the application down into small, loosely coupled, independently deployable services that communicate over APIs. Microservices offer better scalability, fault isolation, and technology flexibility.

```text
[Diagram 2: Monolith vs Microservices]
[Source: internet]
MONOLITH: [ UI + Business Logic + Data Access ] ---> [ Single Database ]

MICROSERVICES:
[ UI ] ---> [ API Gateway ] ---> [ Service A ] ---> [ DB A ]
                            ---> [ Service B ] ---> [ DB B ]
                            ---> [ Service C ] ---> [ DB C ]

```

**3. What is the purpose of the Terraform state file (`terraform.tfstate`)?**

* **Company:** Microsoft, Infosys
* **Answer:** The state file acts as a database that maps the resources defined in your `.tf` configuration files to the real-world resources provisioned in the cloud. Terraform uses this file to detect "drift" (changes made outside of Terraform), determine dependencies, and calculate the exact differences needed during a `terraform plan` or `terraform apply`.


**4. How do you manage infrastructure across multiple environments (e.g., Dev, QA, Prod) using Terraform?**

* **Company:** IBM, Google
* **Answer:** There are two primary ways: using **Terraform Workspaces** or using **Directory Layouts** (folder structures). Workspaces allow you to use the same configuration files with different state files managed dynamically. Directory layouts involve separating configurations into distinct folders (e.g., `envs/dev`, `envs/prod`) using modularized code, which provides stricter isolation and is generally preferred for production setups.

```text

 │    ├── qa/  (calls modules with qa variables)
 │    └── prod/(calls modules with prod variables)

```

* **Company:** Wipro, Amazon

**6. What is an API Gateway in a Microservices architecture?**

* **Company:** Accenture, TCS
* **Answer:** An API Gateway is a server that acts as the single entry point into the system for all clients. It receives requests, routes them to the appropriate underlying microservices, gathers the results, and returns them to the client. It handles cross-cutting concerns like authentication, SSL termination, rate limiting, and request logging.

**7. Explain the concept of Serverless computing.**

* **Company:** Microsoft, Cognizant
* **Answer:** Serverless computing is a cloud execution model where the cloud provider automatically provisions, scales, and manages the infrastructure required to run the code. Developers only focus on writing functions (e.g., Azure Functions, AWS Lambda). Billing is based purely on execution time and resource consumption during execution, not on pre-allocated capacities.

```text
       |
       v
[ Resources spin down instantly ]

```

**8. What are Terraform modules and why are they used?**

* **Company:** Infosys, Google
* **Answer:** A Terraform module is a container for multiple resources that are used together. Modules allow you to encapsulate and reuse infrastructure configurations. Instead of rewriting identical code for network creation, you create a network module and call it whenever needed with different input variables. This keeps code DRY (Don't Repeat Yourself) and easier to maintain.

```text
 +-----------+ +-----------+

```

**9. How do you handle secrets securely in Terraform?**

* **Company:** IBM, Wipro
* **Answer:** Secrets should never be hardcoded into `.tf` files or committed to version control. They should be managed using secure variable passing (e.g., `TF_VAR_` environment variables), injected during CI/CD pipelines, or fetched dynamically using data sources integrated with secret management tools like Azure Key Vault, HashiCorp Vault, or AWS Secrets Manager.


**10. Explain MLOps/LLMOps pipeline integration with DevOps.**

* **Company:** Amazon, Microsoft
* **Answer:** MLOps (Machine Learning Operations) merges ML engineering with DevOps. An MLOps pipeline automates the lifecycle of AI models: gathering data, training/tuning the model, containerizing the trained model (often using Docker), and deploying it to an orchestration platform (like Kubernetes or an Agentic server). CI/CD pipelines are adapted to trigger not just on code changes, but on data or model drift.
