### 2.0 Detailed Lecture Notes: Security Groups & Web Servers

#### 2.1 Establishing Remote Connection to the Linux VM

2.1.1 Before installing any software, an administrator must establish a secure remote session with the **Linux VM** operating system.
2.1.2 Open a local terminal application, such as **Windows PowerShell**, Command Prompt, or a macOS/Linux Terminal.
2.1.3 Locate the **Public IP address** of the deployed Linux VM by navigating to the VM's overview dashboard inside the **Azure Management Portal**.
2.1.4 Execute the standard connection command using the exact administrative **username** defined during the initial VM creation wizard.
2.1.5 The system will present a cryptographic fingerprint warning; the administrator must type `yes` to permanently accept the host's authenticity identity.
2.1.6 Enter the user **password**. *Important Note:* Linux security protocols hide keystrokes entirely; the screen will not display asterisks or dots while typing the password.

2.1.7 **Code Snippet: SSH Connection Command**

```bash
ssh devopsadmin@4.204.224.131
```

#### 2.2 Installing the Nginx Web Server Application

2.2.1 Immediately after successfully logging into the Linux environment, the local package directory must be refreshed to ensure the system downloads the latest software versions.
2.2.2 Execute the package update command: `sudo apt update`. The `sudo` prefix grants temporary root administrator privileges.
2.2.3 Once the update process completes successfully, trigger the installation of the web server by running the command: `sudo apt install nginx`.
2.2.4 The terminal will pause and ask for confirmation to consume disk space; the administrator must type `Y` and press Enter to finalize the installation.
2.2.5 **Important (Asked in multiple interviews):** The Nginx service is programmed by default to listen for web traffic on **Port 80 (HTTP)**. However, installing the software does not automatically open cloud network firewalls.

2.2.6 **Code Snippet: Update and Install Sequence**
**Label Source:** Video Lecture

```bash
sudo apt update
sudo apt install nginx -y

```

#### 2.3 The Cloud Firewall Blockage (Understanding Connection Timeouts)

2.3.1 To test the new web server, copy the VM's **Public IP address** and paste it directly into the address bar of a local web browser (like Chrome or Edge).
2.3.2 The web browser interface will spin continuously and eventually fail, displaying a message such as "This site can't be reached" or **ERR_CONNECTION_TIMED_OUT**.
2.3.3 This failure is intentional and occurs because the **Azure Network Security Group (NSG)** acts as a relentless virtual security guard protecting the VM from unauthorized internet traffic.
2.3.4 When the VM was created, Azure applied a default security rule named **DenyAllInBound** (assigned the lowest priority rank of **65500**).
2.3.5 This default rule explicitly drops all inbound requests originating from the outside internet, including web browsing requests targeting **Port 80**.

#### 2.4 Reconfiguring the NSG to Permit HTTP Traffic

2.4.1 To resolve the connection timeout, the administrator must return to the **Azure Management Portal** and open the settings panel for the specific virtual machine.
2.4.2 Locate and click on the **Networking** menu option on the left-hand navigation bar, which exposes the attached NSG rules.
2.4.3 In the **Inbound port rules** section, click the button labeled **+ Add inbound port rule**.
2.4.4 Leave the Source, Source port ranges, and Destination fields set to their default wildcard values (usually represented as `Any` or `*`).
2.4.5 Locate the **Destination port ranges** text box and change the value explicitly to **80**.
2.4.6 Verify that the network **Protocol** is set to **TCP** and that the final execution **Action** is set to **Allow**.
2.4.7 **Important (Asked in multiple interviews):** Assign a **Priority number** that is lower than 65500 (for example, **310**). The Azure firewall evaluates rules from the lowest number to the highest number. A rule with priority 310 is read before the default block rule at 65500.
2.4.8 Enter a clear, descriptive **Name** for the rule, such as `Allow_HTTP_Traffic`, and click the **Add** button.
2.4.9 Wait approximately 10 to 30 seconds for the portal to confirm the rule has been successfully propagated to the cloud network hardware.
2.4.10 Return to the stalled web browser tab and refresh the page. The firewall block is now lifted, and the **"Welcome to nginx!"** default webpage will render successfully on the screen.

#### 2.5 Architectural Preview: Moving Toward Enterprise Standards

2.5.1 The trainer emphasizes that manually clicking through the portal to assign public IP addresses and open firewall ports directly on a single VM is meant strictly for foundational learning.
2.5.2 In real-world enterprise production environments, individual backend compute servers are almost never assigned direct **Public IPs**.
2.5.3 Instead, large-scale internet traffic is routed through centralized hubs like an **Azure Application Gateway** or an **Azure Load Balancer**, which safely manage public connections and distribute traffic to hidden internal VMs.
2.5.4 Ultimately, all manual portal clicking must be replaced by **Infrastructure as Code (IaC)** tools (like Terraform or ARM templates) that programmatically deploy VNets, subnets, VMs, and NSG rules in a reliable, repeatable manner.

### 4.0 Top 10 Enterprise Interview Questions and Answers

4.1 **Company: Microsoft**
4.1.1 **Question:** Explain the operational difference between attaching an NSG to a **Subnet** versus attaching an NSG directly to a **Network Interface Card (NIC)**.
4.1.2 **Interview Ready Answer:** Attaching an NSG to a **Subnet** creates a security perimeter that automatically applies to all resources placed inside that specific network segment. Attaching an NSG directly to a **NIC** creates a localized micro-firewall that only protects that single specific virtual machine. If both are used simultaneously, incoming traffic is evaluated first by the Subnet NSG, and if allowed, it is evaluated a second time by the NIC NSG. Both perimeters must explicitly allow the traffic for the connection to succeed.

**Label Source:** Internet

4.2 **Company: Amazon (AWS/Azure Multi-Cloud Role)**
4.2.1 **Question:** In the Azure networking ecosystem, what is the default behavior of **outbound** network traffic for a newly created Virtual Machine?
4.2.2 **Interview Ready Answer:** By default, Azure automatically provisions an invisible outbound security rule named **AllowInternetOutBound** (assigned to priority **65001**). This rule explicitly permits the virtual machine to initiate outbound connections to the external internet, which allows the server to automatically download operating system updates and patches without requiring the administrator to write custom outgoing firewall rules.

**Label Source:** Internet

4.3 **Company: TCS**
4.3.1 **Question:** If an NSG contains an "Allow" rule for Port 80 configured at **priority 200**, and a "Deny" rule for Port 80 configured at **priority 150**, what happens when a user attempts to view the website?
4.3.2 **Interview Ready Answer:** The user's web request will be completely blocked, and the website will not load. Azure NSGs process rules in strict numerical order, starting from the lowest priority number and moving upwards. Because **150** is lower than **200**, the firewall evaluates the "Deny" rule first. Upon finding a match, the firewall immediately drops the traffic and halts all further evaluation, meaning the "Allow" rule at 200 is completely ignored.

4.4 **Company: Infosys**
4.4.1 **Question:** What is the purpose of **Augmented Security Rules** within Azure Network Security Groups?
4.4.2 **Interview Ready Answer:** **Augmented Security Rules** are designed to drastically simplify firewall management in complex enterprise environments. Instead of creating dozens of separate, individual rules for every single server IP address, an augmented rule allows an administrator to define multiple destination ports, multiple explicit IP addresses, and specific Azure service tags all combined within one single, highly condensed security rule.

**Label Source:** Internet

4.5 **Company: Wipro**
4.5.1 **Question:** How would an architect securely manage a Linux Virtual Machine without exposing the **SSH Port (22)** to the entire public internet?
4.5.2 **Interview Ready Answer:** The most secure method is to completely remove the Public IP address from the Linux virtual machine. Access is then handled by deploying **Azure Bastion**, a fully managed platform-as-a-service (PaaS) that provides secure and seamless RDP and SSH connectivity directly from the Azure portal over an encrypted **TLS connection (Port 443)**, completely eliminating the risk of public internet port scanning.

**Label Source:** Internet

4.6 **Company: Cognizant**
4.6.1 **Question:** Is it technically possible to use a Network Security Group to filter and block traffic between two virtual machines that reside entirely inside the exact same subnet?
4.6.2 **Interview Ready Answer:** Yes, it is entirely possible. While the default **AllowVnetInBound** rule permits free communication across the entire virtual network, an administrator can attach specific NSGs directly to the individual **NICs** of the VMs. By creating custom "Deny" rules with priority numbers lower than the default rules, the administrator can explicitly block specific ports or IP addresses from communicating with one another, even if they share the identical subnet space.

**Label Source:** Internet

4.7 **Company: Accenture**
4.7.1 **Question:** Describe the functionality of an **Application Security Group (ASG)** and how it directly integrates with standard NSGs.
4.7.2 **Interview Ready Answer:** An **Application Security Group (ASG)** allows administrators to manage network security based on the actual application workload rather than memorizing static IP addresses. By grouping VMs into logical categories (for example, tagging servers as "Frontend-Web-Servers" or "Backend-Database-Servers"), the administrator can write a single NSG rule that applies to the entire ASG name. If a new server is deployed and added to the ASG, it automatically inherits the security rules without any manual IP address updates.

**Label Source:** Internet

4.8 **Company: IBM**
4.8.1 **Question:** What are **default security rules** in an NSG, and what restrictions apply to modifying them?
4.8.2 **Interview Ready Answer:** Every new NSG is automatically populated with a set of default security rules designed to ensure basic infrastructure functionality, such as allowing internal VNet communication and permitting health probes from Azure Load Balancers. These default rules are assigned very high priority numbers (specifically **65000, 65001, and 65500**). The primary restriction is that these default rules cannot be deleted or directly edited; they can only be overridden by creating new, custom rules with lower priority numbers.

4.9 **Company: Capgemini**
4.9.1 **Question:** After an administrator updates or adds an NSG rule, approximately how long does it take for those changes to actively impact network traffic flows?
4.9.2 **Interview Ready Answer:** When an NSG rule is modified, Azure processes and pushes the changes to the underlying host infrastructure extremely rapidly, typically within **10 to 60 seconds**. However, because Azure NSGs operate as **stateful** firewalls, any currently active, established network connections might not be dropped instantly if a new "Deny" rule is applied. The new rule will immediately block all *new* connection attempts, but existing sessions may persist until they naturally time out or are manually terminated.

**Label Source:** Internet

4.10 **Company: HCL**
4.10.1 **Question:** Explain the troubleshooting steps required if an NSG rule is perfectly configured to allow Port 80 traffic, yet the web application remains completely unreachable from the internet.
4.10.2 **Interview Ready Answer:** If the cloud-level NSG is verified as open, the issue lies deeper within the system. First, I would securely SSH into the Linux OS and verify that the internal operating system firewall (such as **UFW** on Ubuntu or **firewalld** on RHEL) is not independently blocking Port 80. Second, I would execute service status commands (like `systemctl status nginx`) to confirm the web server software hasn't crashed. Finally, I would use network diagnostic tools (like `netstat -tulpn`) to verify that the Nginx application is actively bound to and listening on Port 80.

### 5.0 Expert Tips: Perspective from a 20-Year DevOps Architect

5.1 **Mandatory Infrastructure as Code (IaC) Adoption:**
5.1.1 In my two decades of architecture experience, I can tell you that manually clicking through the Azure Portal to configure NSGs and VMs is exclusively for day-one learning.
5.1.2 In a production environment, manual configurations inevitably lead to human error, security breaches, and untrackable changes.
5.1.3 You must immediately begin learning tools like **Terraform** or **Bicep**. Your entire network topology, including every firewall rule, should be written in code, stored in a Git repository, and deployed through automated CI/CD pipelines.

5.2 **Adopt a Zero Trust Network Access Model:**
5.2.1 Never, under any circumstances, leave management ports like **SSH (22)** or **RDP (3389)** permanently open to the public internet (`Source: Any`), even for non-production environments.
5.2.2 Automated botnets scan the entire IPv4 internet address space continuously. A publicly exposed SSH port will face brute-force credential attacks within minutes of deployment.
5.2.3 Always utilize **Azure Bastion** for remote access, or implement **Just-In-Time (JIT) VM Access** via Microsoft Defender for Cloud to open ports only dynamically for authenticated users for a strict, limited timeframe.

5.3 **Modularity and Blast Radius Containment:**
5.3.1 Do not attempt to manage a massive, single Network Security Group containing hundreds of rules applied globally across an entire virtual network.
5.3.2 Design your security architecture using strict **Micro-segmentation**. Separate your frontend web servers, backend application logic, and database clusters into isolated subnets.
5.3.3 Apply dedicated, modular NSGs to each specific subnet tier. By doing this, if a public-facing web server is compromised, the attacker is physically trapped within that subnet because the database subnet's NSG explicitly blocks all traffic originating from the compromised tier.
