A **Landing Zone** is the baseline environment that hosts workloads in a production-ready infrastructure. The trainer breaks down the foundation into 8 core architectural pillars:

8. **High Availability (HA) and Scalability**: Engineering multi-layered structural redundancy to ensure zero downtime.

### Phase B: On-Premises to Cloud Migration Strategy

Before orchestrating any migration, an **Architect** must run a structured framework split into two critical execution gates:

#### 1. Discovery & Assessment Phase

You cannot move what you don't map. An architect must analyze the client's current estate to answer:

* **Business Parameters**: Target cloud selection based on **Cost Parameters**, SLA benchmarks, project delivery timelines, deployment downtime agreements, budget lines, and governance **Compliances** (such as **HIPAA**, **GDPR**, or **RBI compliance**).
* **Technical Footprint Mapping**:
* **Computing Workloads**: Exact counting of physical/virtual hardware, CPU configurations, memory constraints, and host operating systems (**Windows Server**, **Linux** configurations).
* **Storage Matrix**: Evaluating on-premises block and file allocations (**SAN** vs. **NAS** architecture).
* **Database Footprint**: Tracking schema variants, engine versions, and database clusters (**Oracle**, **MySQL**, **SQL Server**).
* **Application Profile**: Categorizing the structural design pattern (**Monolithic Architecture** vs. **Microservices**).
* **Networking and Edge Infrastructure**: Discovering active IP addresses, private tunneling layouts (**VPN** paths), perimeter hardware devices (**Firewall** parameters), and directory structures (**Active Directory** setups).



#### 2. Migration Execution Strategies (The 3 Rs Explained)

* **Lift and Shift (Rehost)**: The fastest execution path. Moving workloads from on-premises straight into cloud compute nodes without code base transformations.
* **Replatform**: Applying minor optimization passes to leverage native platform features without altering the core software engine.
* **Refactor / Rearchitect**: Re-engineering legacy monolithic systems into cloud-native microservices for maximized scalability.

> **Important Operational Rule**: Never migrate mission-critical systems first. Always arrange migrations hierarchically from **Least Critical** to **Most Critical**. Moving low-risk testing nodes early builds operational confidence and framework validation before moving core production nodes.

### Phase D: Deep-Dive into Azure Virtual Network (VNet) & Networking Basics

To run a 3-tier production app (Frontend, Backend, Database), native compute systems must reside inside an explicit corporate network bubble.

#### 1. Core Functions of a Router

* **IP Assignment**: Dynamically hands unique routing addresses to every registered network participant.
* **Communication Bridging**: Orchestrates the communication loops between internal machines while translating data boundaries.

#### 2. The CIDR (Classless Inter-Domain Routing) Formulation

When writing individual comma-separated IP matrices becomes impossible at enterprise scale, engineers utilize **CIDR Notation**.

* **Formula for Available Addresses**: 
$$2^{(32 - n)}$$


 (where $n$ represents the routing suffix).
* **Sample Calculation ($\text{/24}$ block)**:

$$32 - 24 = 8$$


$$2^8 = 256 \text{ absolute IP allocations}$$



#### 3. Step-by-Step VNet Provisioning in Azure Portal

1. Select the destination **Subscription** block and reference an active **Resource Group** (e.g., `rg-networking`).
2. Input the customized virtual network name identifier (e.g., `dlink`).
3. Allocate your global geography parameters (**Region** block selection like `Canada Central`).
4. Define your core Address Space block inside the networking configuration window (e.g., configuring `10.143.0.0/16`).
5. Choose **Review + Create**, pass validation checkpoints, and confirm building the infrastructure node.

### Phase E: Provisioning Compute Infrastructure & Designing High Availability

When building an internal server structure within the portal interface, specific properties must be designated:

#### 1. Compute Configuration Properties

* **Subscription & Resource Group Alignment**: Grouping resources safely under defined asset boundaries (e.g., `rg-dev`).
* **Compute Identity Designation**: Defining names for nodes (e.g., `netflix-vm`).
* **Operating System Target (Image)**: Selecting your OS kernel environment (e.g., **Ubuntu Server 24.04 LTS**, **Red Hat Enterprise Linux**, or **Windows Server**).
* **Compute Instance Profile (Size Selection)**: Balancing hardware tiers against standard pricing lines. Tiers include:
* **General Purpose Tiers (D-Series)**: Well-balanced setups for testing or normal standard apps.
* **Compute Optimized Instances (F-Series)**: High-performance configurations for raw task execution processing.
* **GPU Intensive Arrays (N-Series)**: High-cost graphic nodes meant for complex virtualization layouts.
* **Burstable Low-Tier Variants (B-Series)**: Cheap configurations tailored for lightweight non-continuous utility code tasks.



#### 2. Designing Redundancy Zones (Mitigating Infrastructure Faults)

The trainer warns against trusting single-point configurations. Redundancy strategy profiles include:

```
                          [ Canada Central Region ]
                                     |
         ---------------------------------------------------------
        |                            |                            |
  [ Availability Zone 1 ]      [ Availability Zone 2 ]      [ Availability Zone 3 ]
   (Isolated Data Center)       (Isolated Data Center)       (Isolated Data Center)
        |
   -------------------------------------------------
  |                 |                               |
[ Fault Domain 1 ] [ Fault Domain 2 ]             [ Fault Domain 3 ]
 (Server Rack 1)    (Server Rack 2)                (Server Rack 3)
 Powered by:        Powered by:                    Powered by:
 MPPKVVCL           Adani Power                    Tata Power

```

* **Availability Zones**: Isolated physical data center clusters within a native cloud region. Spreading workloads across Zones safeguards systems from complete local building failures (such as structural bomb blasts, fires, or utility drop-offs).
* **Availability Sets & Fault Domains**: Organizing virtual machines across separate physical hardware racks (**Fault Domains**) within a single data center facility. Each rack relies on isolated power bus lines and grid distributors (analogous to utility feeds like **MPPKVVCL**, **Adani**, or **Tata Power**). If a single rack power system fails, alternate arrays keep your systems active.

## 4. Industry Interview Prep Kit (Cross-Company Verification)

### Question 1 (Asked by: Tata Consultancy Services)

**How do you differentiate between an Azure Region, an Availability Zone, and an Availability Set?**

* **Answer**: An **Azure Region** is a geographical area containing multiple data centers connected through a low-latency network. An **Availability Zone** is an isolated physical location *within* a region, offering independent power, cooling, and network lines to prevent complete data center failures. An **Availability Set** is a logical grouping *within* a data center that spreads VMs across multiple **Fault Domains** (racks) to protect against localized hardware failures.
* **Diagram**:

```
[ Geographical Region ]
   ├── [ Availability Zone 1 (Data Center Node) ] -> Set up with Availability Set (Rack 1 / Rack 2)
   └── [ Availability Zone 2 (Data Center Node) ] -> Set up with Alternate Virtual Target Pools

```

### Question 2 (Asked by: Wipro)

**What are Fault Domains and Update Domains within an Azure Availability Set environment?**

* **Answer**: **Fault Domains** represent distinct physical server racks sharing common power modules and physical network switches. Azure allocates VMs across separate fault domains to avoid localized hardware failures. **Update Domains** are logical boundaries used during platform updates, ensuring that only one domain group is rebooted at a time to maintain application availability.
* **Diagram**:

```
[ Availability Set Boundary ]
   ├── Rack 1 [Fault Domain 0 / Update Domain 0] -> Host Instance Server Node 1
   └── Rack 2 [Fault Domain 1 / Update Domain 1] -> Host Instance Server Node 2

```

### Question 3 (Asked by: Cognizant)

**How many total IP addresses are available in a `/24` Azure sub-network block? Explain the allocation parameters.**

* **Answer**: A standard `/24` block provides $2^{(32-24)} = 256$ total IP addresses. However, Azure reserves **5 IP addresses** for internal operations: `.0` (Network address), `.1` (Default Gateway), `.2`/`.3` (Azure DNS Mapping space), and `.255` (Network Broadcast target). This leaves **251 usable IP addresses** for user resources.
* **Diagram**:

```
[ Subnet Range Block IP Segment Matrix Pool ]
   ├── Reserved: .0, .1, .2, .3, .255  (System Management Spaces)
   └── Usable Pool: .4 through .254    (Assigned to Virtual Interface Tiers)

```

### Question 5 (Asked by: Infosys)

**Explain the core architectural components of the Hub-and-Spoke networking design model.**

* **Answer**: The **Hub-and-Spoke** topology uses a central networking node (the Hub) to host shared infrastructure assets such as express routes, perimeter firewalls, and hybrid gateways. Isolated application workloads reside in peripheral virtual networks (the Spokes), which peer directly with the central hub to access shared resources securely.
* **Diagram**:

```
       [ Spoke VNet: App 1 ]
               │
[ Spoke VNet: App 2 ] ─── [ Central Hub VNet (Firewall/Gateway) ] ─── [ On-Premise Network Space ]
               │
       [ Spoke VNet: App 3 ]

```

### Question 6 (Asked by: HCLTech)

**What are the primary differences between Rehosting, Replatforming, and Rearchitecting migration strategies?**

* **Answer**: **Rehosting (Lift-and-Shift)** moves virtual systems directly to cloud platforms without making code modifications. **Replatforming (Lift-Tweak-and-Shift)** applies minor optimization adjustments, such as moving a database into a managed cloud database service, without changing the application's core logic. **Rearchitecting (Refactoring)** involves breaking down monolithic applications into microservices to maximize cloud-native capabilities.
* **Diagram**:

```
Legacy System ──► [ Replatform ] ──► Managed PaaS Cloud Engine
Legacy System ──► [ Refactor ] ──► Container Kubernetes Cloud Matrix

```

### Question 8 (Asked by: Tech Mahindra)

**When should you select Azure burstable B-Series compute instances over standard D-Series compute arrays?**

* **Answer**: **B-Series instances** are ideal for workloads with low baseline CPU utilization that experience occasional spikes in demand, such as development code environments, small database engines, or monitoring tools. They accumulate CPU credits during quiet periods to handle performance bursts, making them a cost-effective alternative to standard **D-Series instances**, which provide consistent CPU performance for sustained production workloads.
* **Diagram**:

```
B-Series:  Baseline Util ─── Sustained Credits Pool Accrual ───► CPU Spike Performance Burst Allowed
D-Series:  Sustained Performance Tiers Matrix (Continuous Flat Line Target Tiers)

```

This metadata helps determine proper cloud compute sizing, estimate monthly operational costs, and execute replication pipelines directly into Azure storage layers.

### Question 10 (Asked by: Deloitte)

**How do you approach a situation where an assigned cloud network CIDR block overlaps with an active on-premises private network address space?**

* **Answer**: Overlapping IP space prevents proper routing between on-premises environments and the cloud. To resolve this, you must re-architect the landing zone's subnet layouts using a distinct, non-overlapping private address range. If address spaces cannot be changed, you must implement Network Address Translation (NAT) rules or configure intermediate proxy proxy routing spaces to rewrite source and destination IP coordinates during data transit.
* **Diagram**:

```
[ On-Premise: 10.0.0.0/16 ] ──► [ NAT Network Mapping Translation Proxy Gateway ] ──► [ Azure VNet Space: 10.0.0.0/16 Pool ]

```

## 5. Devops Architect Tips Section

Assuming the role of a **DevOps Architect with 20+ years of cloud and infrastructure experience**, here are key recommendations for production environments:

1. **Enforce Infrastructure as Code (IaC) and Prevent Manual Portal Creation**:
Manual configurations in the public cloud lead to configuration drift and security inconsistencies. All production environments, VNets, and subnets should be written using declarative tools such as **Terraform**, **OpenTofu**, or **Azure Bicep**. Infrastructure code must be managed in a version control system (Git) and deployed via strict **CI/CD pipelines** (e.g., GitHub Actions or GitLab CI) with linting, security scans, and mandatory peer approval stages.
2. **Establish Strict IP Address Allocation Strategies**:
Many enterprise cloud migrations encounter roadblocks due to poorly planned IP address structures. Do not provision large subnet blocks (like a `/16`) for a single cluster or application zone. Instead, implement a centralized address management strategy that utilizes tight, scalable CIDR subdivisions. Ensure all address structures are accounted for across all corporate data environments to allow for seamless integration with multi-cloud resources and corporate **Hub-and-Spoke VNet peerings** without routing conflicts.
3. **Incorporate Cross-Zone Load Balancing and Auto-Scaling Metrics**:
Provisioning multiple VMs across different availability zones is only effective if your applications are designed to fail over gracefully. Always place compute resources behind network traffic managers, such as **Azure Load Balancers** or **Application Gateways**, configured with cross-zone traffic routing rules. Combine these setups with dynamic health probe rules and automated instance group scaling metrics (**Virtual Machine Scale Sets**) to ensure the platform handles traffic variations smoothly and remains resilient during unforeseen cloud hardware failures.

### Sample Code Blocks for Quick Reference

#### 1. Azure CLI: Provisioning VNet and Compute Infrastructure

```bash
# Create a Resource Group for the environment
az group create --name rg-dev-canada --location canadacentral

# Provision the Virtual Network with defined CIDR blocks
az network vnet create \
  --resource-group rg-dev-canada \
  --name dlink-vnet \
  --address-prefixes 10.143.0.0/16 \
  --subnet-name frontend-subnet \
  --subnet-prefixes 10.143.1.0/24

# Create a Virtual Machine instance within the subnet
az vm create \
  --resource-group rg-dev-canada \
  --name netflix-vm \
  --image Ubuntu2204 \
  --vnet-name dlink-vnet \
  --subnet frontend-subnet \
  --size Standard_D2s_v3 \
```
