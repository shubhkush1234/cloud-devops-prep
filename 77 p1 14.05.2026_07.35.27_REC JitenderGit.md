## Interview Questions & Answers

*The following questions are compiled from common technical interviews across major technology firms.*

### 1. What is the difference between `git merge` and `git rebase`? (Google, Meta) **[Important]**

* **Answer:** `git merge` takes the contents of a source branch and integrates them with a target branch, creating a new "merge commit." `git rebase` rewrites the project history by moving the sequence of commits to a new base commit, creating a linear history.
* **Diagram:**

```text
Merge:
A---B---C (main)
     \
      D---E (feature)
      |
A---B---C---F (main + feature)

Rebase:
A---B---C (main)
         \
          D'---E' (feature)

```

* **Source:** Internet

### 2. Explain the "Detached HEAD" state. (Amazon/AWS)

* **Answer:** This state occurs when you checkout a specific commit hash rather than a branch. You can make commits, but they aren't part of any branch and will be lost if you switch to another branch without creating a new one.
* **Diagram:**

```text
(HEAD) -> Commit 3 -> Commit 2 -> Commit 1
          (Not on any branch)

```

* **Source:** Internet

### 3. How do you resolve a merge conflict? (Red Hat) **[Important]**

* **Answer:** When Git cannot automatically combine changes, it marks the file with conflict markers (`<<<<<<<`, `=======`, `>>>>>>>`). You must manually edit the file to choose the desired code, remove the markers, run `git add` to stage the resolution, and then `git commit`.
* **Diagram:**

```text
<<<< HEAD (Local)
Code A
=======
Code B
>>>> (Remote)

```

* **Source:** Internet

### 4. What is the difference between `git reset` and `git revert`? (Microsoft)

* **Answer:** `git reset` moves the branch pointer backward, effectively erasing history (dangerous for shared branches). `git revert` creates a *new* commit that undoes the changes of a previous commit, maintaining a safe history.
* **Diagram:**

```text
Reset: [C1]--[C2]--[C3]  -> [C1] (Pointer moves back)
Revert: [C1]--[C2]--[C3]--[C2_inverse] (New commit added)

```

* **Source:** Internet

### 5. What is a "Bare" repository? (IBM)

* **Answer:** A bare repository does not have a working directory. It contains only the `.git` version control data. It is used as a central hub for sharing code, and developers never commit directly to it.
* **Diagram:**

```text
[Working Dir] <-> [Non-Bare Repo] <-> [Bare Repo]

```

* **Source:** Internet

### 6. What is `git cherry-pick`? (Oracle)

* **Answer:** It allows you to select an arbitrary commit from one branch and apply it to another without merging the entire branch.
* **Diagram:**

```text
Branch A: [C1]--[C2]--[C3]
                    \
Branch B: [C1]--------[C2']

```

* **Source:** Internet

### 7. What does `git fetch --all` do? (VMware)

* **Answer:** It updates your local copy of *all* remote tracking branches for all remotes defined in your configuration, allowing you to see what has changed upstream without altering your current working files.
* **Diagram:**

```text
Remote -> [Fetch] -> Local Tracking Branches (origin/main, origin/feat)

```

* **Source:** Internet

### 8. Explain the Git Stash command. (Stripe)

* **Answer:** `git stash` temporarily shelves (saves) changes you've made to your working directory so you can work on something else, and then come back and re-apply them later.
* **Diagram:**

```text
Working Dir -> [Stash] -> (Stack: stash@{0})

```

* **Source:** Internet

### 9. What is a "Git Hook"? (LinkedIn)

* **Answer:** Git hooks are scripts that trigger automatically at specific points in the Git lifecycle (e.g., `pre-commit`, `post-receive`). They are often used to enforce code standards or run tests before a commit is allowed.
* **Diagram:**

```text
[Code] -> [pre-commit hook] -> [Testing/Linting] -> [Commit Success/Fail]

```

* **Source:** Internet

### 10. Describe the Git object model. (General Advanced)

* **Answer:** Git stores data as a series of objects: *Blobs* (file content), *Trees* (directory structures), *Commits* (snapshots of trees with metadata), and *Tags* (pointers to specific commits).
* **Diagram:**

```text
Commit Object -> points to -> Tree Object -> points to -> Blob Object

```

* **Source:** Internet

## DevOps Architect Tips

*Advice based on 20+ years of industry experience:*

1. **Prioritize Trunk-Based Development:** Avoid long-lived feature branches. They lead to "merge hell." Merge small, incremental changes to `main` as frequently as possible to keep your code integrated and reduce risk.
2. **Never Hardcode Secrets:** Your `.gitignore` is your first line of defense. Use environment variables or secret management tools (like HashiCorp Vault or AWS Secrets Manager). Hardcoding credentials in Git is a major security vulnerability.
3. **Automate Everything:** If you find yourself typing the same complex Git command more than three times, write an alias or a shell script. Git efficiency is about speed and reducing human error in CI/CD pipelines.
