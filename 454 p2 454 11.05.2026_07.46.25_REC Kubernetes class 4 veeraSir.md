# Study Guide & Technical Documentation: Kubernetes Cluster Autoscaler Deployment

---

## To-the-Point Summary

This lecture details the practical implementation and configuration of the **Cluster Autoscaler** on an Amazon EKS cluster. While the Horizontal Pod Autoscaler (HPA) manages the number of pod replicas based on workload metrics, it is constrained by the physical capacity of the cluster's worker nodes. The Cluster Autoscaler acts as the infrastructure scaling mechanism. When pods enter a `Pending` state due to insufficient CPU or memory resources on existing nodes, the Cluster Autoscaler communicates with the cloud provider (AWS Auto Scaling Groups in this case) to provision new compute instances. The session covers deploying the official manifest, resolving configuration errors (like cluster name mismatch), configuring necessary IAM permissions, setting Node Group scaling limits, and validating the scale-up process under simulated load.

---

## Detailed Lecture Notes & Configuration Steps

### 1. Introduction to Cluster Autoscaler

When HPA dynamically increases the replica count of a deployment, the cluster may run out of hardware resources (CPU/Memory). When this happens, new pods cannot be scheduled and remain in a `Pending` state. The **Cluster Autoscaler** monitors for these specific `Pending` pods and triggers an infrastructure scale-up.

> **[IMPORTANT]** HPA scales the *applications* (Pods), while the Cluster Autoscaler scales the *infrastructure* (Nodes).

### 2. Deploying the Cluster Autoscaler Manifest

The first step is to apply the official Cluster Autoscaler manifest provided by Kubernetes for your specific cloud provider.

1. **Apply the Manifest:**

### 3. Configuring IAM Permissions for the Autoscaler

For the Cluster Autoscaler to manipulate AWS resources (specifically Auto Scaling Groups), it must possess the correct IAM permissions.

1. Navigate to the **AWS IAM Console**.
2. Locate the IAM Role attached to your EKS Node Group.
3. Attach a policy that grants permissions to describe and modify ASGs.
*Trainer Note: For demonstration, `AdministratorAccess` was attached, but in production, a strict inline policy should be used containing actions like `autoscaling:SetDesiredCapacity` and `autoscaling:TerminateInstanceInAutoScalingGroup`.*

[Screenshot of AWS IAM Console showing permissions attached to the Node Group role]

### 4. Updating EKS Node Group Scaling Limits

The AWS Auto Scaling Group (ASG) must have bounds defined to allow scaling while preventing unbounded costs.

1. Go to the **Amazon EKS Console** -> Select your cluster -> **Compute** tab.
2. Select your Node Group and click **Edit**.
3. Set the scaling configurations. In the lecture, the trainer sets:
* **Minimum size:** 2
* **Maximum size:** 6

The default manifest contains placeholder values. The autoscaler must be explicitly told which EKS cluster to monitor and scale.

1. **Edit the Deployment:**
```bash
kubectl -n kube-system edit deployment.apps/cluster-autoscaler

```



```
2.  **Modify the Command Arguments:** Locate the `spec.containers.command` section and update the `--node-group-auto-discovery` flag with your exact cluster name.

## Interview Questions & Interview-Ready Answers

### Q1: How does the Kubernetes Cluster Autoscaler fundamentally operate?
**Asked at:** *Amazon, Google* — **[IMPORTANT]**

**Answer:**
The Cluster Autoscaler (CA) continuously monitors the cluster for pods that have failed to schedule and are stuck in a `Pending` state specifically due to insufficient resource requests (CPU or Memory) on the current worker nodes. When it detects such pods, it simulates the addition of a new node. If the simulation proves that a new node would allow the pending pods to schedule, the CA makes an API call to the cloud provider's scaling service (like AWS Auto Scaling Groups) to increase the node count. Conversely, it periodically checks for nodes that have been underutilized for an extended period and safely cordons, drains, and terminates them to save costs.

> | New Worker Node | <------------------------------ |  AWS Auto Scaling  |

> ```

### Q2: What is the distinct difference between HPA, VPA, and Cluster Autoscaler?
**Asked at:** *Microsoft, Cisco*

**Answer:**
These three controllers serve different layers of the scalability stack:
*   **Horizontal Pod Autoscaler (HPA):** Scales the *number of pod replicas* in a deployment based on observed metrics like CPU utilization or custom metrics. It scales the application out.
*   **Vertical Pod Autoscaler (VPA):** Modifies the *resource requests and limits* (CPU/RAM) of existing pods based on historical usage. It scales the application up (making individual pods larger).
*   **Cluster Autoscaler (CA):** Scales the *number of physical or virtual worker nodes* in the cluster. It provides the underlying infrastructure capacity required when HPA or manual scaling demands more resources than are currently available.

### Q3: How does the Cluster Autoscaler decide which node group to scale up when multiple are available?

**Asked at:** *Microsoft, IBM*

**Answer:**
When a cluster has multiple node groups (e.g., one with `t3.medium` instances and another with `m5.large` instances) and a pending pod could fit on either, the CA uses an **Expander** strategy to make the decision. The default expander is `random`. However, you can configure other strategies via the `--expander` flag in the CA deployment:

* `least-waste`: Chooses the node group that will have the least amount of unused CPU/Memory after the pod is scheduled.
* `most-pods`: Chooses the node group that would allow the highest number of pending pods to be scheduled.
* `price`: (Supported by some cloud providers) Chooses the most cost-effective node group.

### Q4: Explain how the Cluster Autoscaler handles node scale-down.
**Asked at:** *Adobe, Red Hat*

**Answer:**
The CA constantly evaluates node utilization based on the requested resources of the pods running on them (not actual usage). By default, if a node's total requested resources fall below 50% for 10 consecutive minutes, it becomes a candidate for scale-down. The CA then simulates moving the remaining pods on that node to other healthy nodes in the cluster. If all pods can be successfully rescheduled elsewhere without violating constraints (like PodAntiAffinity or PodDisruptionBudgets), the CA will cordon the node, evict the pods gracefully, and finally issue a terminate command to the cloud provider.

### Q5: What conditions prevent the Cluster Autoscaler from scaling down a node?

**Asked at:** *Netflix, Salesforce* — **[IMPORTANT]**

**Answer:**
Several constraints act as blockers for node scale-down to ensure workload stability:

1. **Strict PodDisruptionBudgets (PDB):** If evicting a pod would violate a PDB (e.g., `minAvailable: 2` and only 2 exist).
2. **Local Storage:** Pods utilizing emptyDir local storage cannot be safely moved.
3. **Kube-System Pods:** By default, nodes running pods in the `kube-system` namespace (like coreDNS or metric-server) that aren't managed by a DaemonSet will not be scaled down unless explicitly overridden.
4. **No Alternative Capacity:** If the pods on the underutilized node cannot fit onto any other existing nodes.
5. **Specific Annotations:** If a pod has the `cluster-autoscaler.kubernetes.io/safe-to-evict: "false"` annotation, the CA will refuse to move it.

### Q6: How does Cluster Autoscaler integrate securely with AWS EKS?
**Asked at:** *AWS, Uber*

**Answer:**
The Cluster Autoscaler must interact directly with the AWS Auto Scaling API to adjust the desired capacity of the ASGs backing the EKS Node Groups. The secure and recommended way to implement this is using **IAM Roles for Service Accounts (IRSA)**. You create an IAM role with a specific policy allowing `autoscaling:SetDesiredCapacity`, `autoscaling:TerminateInstanceInAutoScalingGroup`, and read permissions. This IAM role is then mapped to the Kubernetes Service Account used by the `cluster-autoscaler` pod via OIDC federation. This ensures that only the autoscaler pod—not the entire worker node—has the permissions to modify infrastructure.

### Q7: If you have an ASG set to Min: 2, Max: 10, but the CA refuses to scale past 5 nodes despite pending pods, what is the most likely cause?

**Asked at:** *Google, Capgemini*

**Answer:**
If the CA logs show it wants to scale but cannot, the issue usually lies outside of Kubernetes in the cloud provider's infrastructure quotas. The most likely causes are:

1. **EC2 Instance Quotas:** The AWS account has hit its vCPU limit for that specific instance type in that region (e.g., hitting the limit for `Standard (A, C, D, H, I, M, R, T, Z) instances`).
2. **IP Address Exhaustion:** The subnets associated with the ASG have run out of available private IP addresses, preventing new EC2 instances or ENIs from being provisioned.

### Q8: Explain the concept of "Overprovisioning" in the context of Kubernetes Autoscaling.

**Asked at:** *Amazon, Adobe*

**Answer:**
Cluster Autoscaler takes time to spin up new EC2 instances (often 2-4 minutes). During this time, sudden spikes in traffic can cause latency as pods wait in `Pending`. **Overprovisioning** mitigates this by running low-priority "Pause Pods" (containers that do nothing but request resources). When a real, high-priority application pod needs to scale up, the Kubernetes scheduler immediately evicts the low-priority Pause Pods to make room. The high-priority app starts instantly. The eviction of the Pause Pods puts *them* into a `Pending` state, which then triggers the Cluster Autoscaler to provision new nodes in the background without affecting application latency.

### Q9: How do you troubleshoot a scenario where Cluster Autoscaler is running, but pending pods are not triggering a scale-up?

**Asked at:** *Cognizant, Deloitte*

**Answer:**
The primary troubleshooting step is to check the Cluster Autoscaler logs: `kubectl logs -f deployment/cluster-autoscaler -n kube-system`.
Key things to look for:

* Are there IAM access denied errors? (Verify IRSA).
* Does the CA see the pending pods?
* Are there `nodeSelector`, `affinity`, or `taint/toleration` mismatches? If a pending pod requires a specific label (e.g., `gpu=true`) but the ASG managed by the CA does not have nodes with that label, the CA will output `pod didn't trigger scale-up: 1 node(s) didn't match Pod's node affinity/selector`.

### Q10: What happens if a deployment has no resource requests defined, but needs to scale?

**Asked at:** *IBM, EY*

**Answer:**
This is a critical failure point. Both HPA and Cluster Autoscaler rely heavily on resource `requests`. HPA needs requests to calculate percentage-based utilization. Cluster Autoscaler uses the sum of `requests` (not actual usage) to determine if a node is full or if it's safe to scale down. If a pod has no resource requests, the scheduler assumes it takes 0 resources and will pack it onto a node. However, this pod can easily cause the node to run out of memory (OOM), leading to node crashes and evictions, bypassing the autoscaling logic entirely.

## DevOps Architect Tips for Production Scaling

1. **Implement Aggressive Overprovisioning for Bursty Workloads:** The lag time of cloud providers provisioning new VMs (2-5 minutes) is unacceptable for highly dynamic, user-facing applications. Use PriorityClasses to deploy "Pause Pods" that reserve 10-15% of your cluster capacity. This allows instant scheduling for critical apps while pushing the infrastructure provisioning delay to the background.
2. **Strictly Define Resource Requests and Limits:** Autoscaling mathematics are entirely broken if workloads do not have accurate resource profiles. Enforce Kubernetes `LimitRanges` at the namespace level to ensure no pod can be deployed without explicit CPU and Memory requests. This guarantees the Scheduler and Cluster Autoscaler have an accurate mathematical model of cluster saturation.
3. **Decouple CA Permissions using IRSA:** Never attach broad EC2 modification permissions to the underlying worker node IAM profile. If a container is compromised, the attacker gains the ability to manipulate your AWS infrastructure. Always utilize IAM Roles for Service Accounts (IRSA) to bind the `autoscaling` policies strictly to the OIDC identity of the `cluster-autoscaler` pod itself.
