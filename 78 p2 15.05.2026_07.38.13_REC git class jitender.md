### **Detailed Notes: Azure Pipeline Architecture & Branching**

#### **1. Pipeline Anatomy: Jobs and Agents**

* **Pipeline Definition:** A sequence of tasks that runs on a specific agent pool.
* **Why use Jobs?**
* **Parallel Execution:** Run multiple tasks simultaneously to save time.
* **Dependency Management:** Control the order of operations (e.g., build must succeed before deploy).
* **Isolation:** Each job runs in its own environment, preventing configuration drift or dependency conflicts between tasks.
* **Log Readability:** Segregating tasks into jobs makes debugging significantly easier.



**Diagram 1: Azure Pipeline Structure**

```text
[Pipeline]
   |
   |-- [Job 1: Build] (Agent Pool: Linux)
   |     |-- Task: Install Terraform
   |     |-- Task: Terraform Init/Plan
   |
   |-- [Job 2: Security] (Agent Pool: Security-Scanner)
   |     |-- Task: SAST Scan (SonarQube)
   |
   |-- [Job 3: Deploy] (Agent Pool: Server-Cluster)
         |-- Task: Terraform Apply

```

#### **2. Branching Strategies**

The trainer emphasized that no single strategy is "best"; it depends on company requirements, but Trunk-based branching is highly recommended for infrastructure.

* **Trunk-Based:** Developers work in short-lived feature branches and merge into `main` frequently.
* **GitFlow:** Uses multiple long-lived branches (main, develop, feature, hotfix). While robust, it can create complexity in CI/CD pipelines.

**Diagram 2: Branching Flow**

```text
Feature Branch -> Pull Request -> Approved -> Merge into Main
                                     |
                          [Triggers CI/CD Pipeline]

```

#### **3. CI/CD Workflow & Security Integration**

* **Testing Steps:**
1. **Code Commits:** Trigger pipeline.
2. **Linting/SAST:** Scan for hardcoded credentials/vulnerabilities (e.g., TFLint, SonarQube).
3. **DAST:** Dynamic scan of the running application.
4. **Deployment:** Apply infrastructure changes.
5. **Rollback Logic:** Use `kubectl rollout undo` or Helm rollback if the deployment fails.

#### **Answers Section**

**Q1 (Important): Explain the difference between SAST and DAST.**

* **Answer:** SAST (Static Application Security Testing) is "white-box" testing; it checks source code for vulnerabilities before the build. DAST (Dynamic Application Security Testing) is "black-box" testing; it tests the application from the outside while it is running.
* **Diagram 3 (SAST vs DAST):**
`Code -> [SAST] -> Build -> Deploy -> [DAST]`

**Q4: How do you implement a rollback in K8s via CI/CD?**

* **Answer:** Use native K8s commands (`kubectl rollout undo deployment/<name>`) or Helm (`helm rollback <release>`). This should be automated in the pipeline if health checks fail post-deployment.
* **Diagram 4 (Rollback Process):**
`Pipeline Trigger -> Deploy -> Health Check Fail -> [Trigger Rollback]`

**Q7 (Important): Explain the Blue-Green deployment strategy.**

* **Answer:** Blue-Green deployment uses two identical production environments. One (Blue) is live, and the other (Green) receives the new version. Once tested, traffic is switched from Blue to Green. This allows for near-zero downtime and instant rollback.
* **Diagram 5 (Blue-Green Flow):**
`Router -> Blue (v1) [Live]`
`Router -> Green (v2) [Idle/Testing]`
*Switch:* `Router -> Green (v2) [Live]`

### **Tips from a DevOps Architect (20+ Years Experience)**

1. **Embrace Idempotency:** Ensure your infrastructure scripts (Terraform/Ansible) can run multiple times without causing side effects. If your pipeline fails halfway through, re-running it should fix the state, not break it.
2. **Shift-Left Everything:** Security, performance testing, and linting should happen as close to the developer's commit as possible. Catching an error in a development branch is 100x cheaper than fixing a breach in production.
3. **Treat Infrastructure as a Product:** Don't just build pipelines; build a **platform**. Create reusable pipeline templates so that every team in your organization doesn't have to reinvent the wheel for standard tasks like building a container or scanning code.
