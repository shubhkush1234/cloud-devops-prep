# GitHub Branch Protection, Code Review, and Terraform CI/CD Pipelines

## 1. To-the-Point Summary

This session covers the end-to-end workflow of managing code changes using Git and GitHub within an enterprise DevOps model. It focuses on shifting away from direct commits on the main branch by implementing **Branch Protection Rules**. Key workflows demonstrated include creating feature branches, tracking modifications/deletions in VS Code, inviting collaborators, and establishing multi-reviewer approval chains. Additionally, the distinction between **Infrastructure pipelines (Terraform CI with `tflint`, `tfsec`, and `plan`)** and **Application deployment pipelines** is highlighted to show how automated gatekeeping ensures environment stability.

### C. Configuring Branch Protection Rules

To enforce enterprise security gates and ensure no code reaches production unreviewed, administrators configure **Branch Protection Rules**:

* **Path:** Repository `Settings` $\rightarrow$ `Branches` $\rightarrow$ Click `Add classic branch protection rule` (or `Add rule`).
* **Branch Name Pattern:** Enter the exact target branch pattern, such as `main`.

#### Key Protection Policies Configured:

1. **Require a pull request before merging:** Enforces that all modifications must arrive through a PR instead of direct commits.
2. **Require Approvals:** Set the minimum number of distinct approvals (e.g., at least 1 or 2 reviewers) required before a PR can merge.
3. **Do not allow bypassing the above settings:** Prevents repository owners, administrators, and DevOps leads from bypassing the code review controls.
4. **Require status checks to pass before merging:** Integrates automated CI checks (e.g., security scanners and validation scripts) that must return a green status before the merge button becomes active.

### D. Automated Infrastructure Pipelines (Terraform Workflow)

When managing Infrastructure as Code (IaC) with Terraform via GitHub Actions, the validation sequence is typically separated into two processing phases:

#### 1. Pull Request Pipeline (CI Gate)

This pipeline triggers automatically on every PR open or update event. It is strictly observational and descriptive; it does not deploy infrastructure.

* **TF Lint / Checkov:** Performs static code analysis and checks for compliance with cloud security policies.
* **Terraform Init:** Standardizes backend initialization and downloads providers.
* **Terraform Validate:** Validates syntax validity.
* **Terraform Plan:** Generates an execution plan detailing exactly what will be added, changed, or destroyed.

#### 2. Deployment Pipeline (CD Run)

Triggers exclusively upon a successful merge into the protected `main` branch.

* Executes `terraform init`, `terraform plan`, and finally **`terraform apply`** to provision the actual cloud resources (e.g., Azure or AWS infrastructure).

```
  [PR Opened] ---> [Run Static Scans (TFLint, tfsec)] ---> [Terraform Plan] ---> [Reviewer Approval]
                                                                                          |
  [Cloud Resources Provisioned] <--- [Terraform Apply] <--- [Merge to main] <------------+

```

## 3. Interview Questions & Ready Answers

### Q1. What is the standard workflow for modifying infrastructure code when the main branch is protected?

**Company asked:** *HCL, Wipro*

**Answer:**

Developers cannot push code straight to a protected branch. The step-by-step workflow requires creating a separate feature branch locally, adding or updating the code, and pushing the feature branch to GitHub. From there, a Pull Request (PR) is opened against the `main` branch. This PR triggers an automated linting and plan pipeline, while notifying assigned peers to conduct a code review and grant the required merge approvals.

```
[Local Repo] --(git checkout -b feature)--> [Make Changes] --(git push)--> [GitHub PR] ---> [Main Branch]

```

### Q2. **[IMPORTANT]** Explain the difference between a Pull Request (CI) pipeline and a Deployment (CD) pipeline in a Terraform context.

**Company asked:** *TCS, Cognizant, Infosys*

**Answer:**

* **PR Pipeline (CI):** Runs on pull requests. It executes formatting, static security scanning (`tfsec`, `checkov`), linting (`tflint`), initialization, and `terraform plan`. It ensures the code is syntactically sound and compliant without changing actual infrastructure.
* **Deployment Pipeline (CD):** Triggers only after code is merged into the default branch (e.g., `main`). It executes `terraform apply` to deploy or modify the targeted cloud infrastructure resources.

### Q4. What happens to a pull request on GitHub if a feature branch is deleted right after a successful merge?

**Company asked:** *Accenture*

**Answer:**

When a PR is successfully merged, its state changes to "Merged and Closed". Deleting the remote feature branch cleanly cleans up the repository's active branch list. The full historical log, commit reference points, differential reviews, and comment trails remain preserved within the closed PR record for auditing purposes.

### Q5. **[IMPORTANT]** How does GitHub's "Require status checks to pass before merging" rule integrate with external CI/CD tools like Azure Pipelines or GitHub Actions?

**Company asked:** *LTIMindtree, IBM*

**Answer:**

This option links code merging permissions to the completion status of continuous integration jobs. When an automated runner initiates a pipeline (e.g., running tests or validation code), it sends a "Pending" status check back to the GitHub commit SHA. Once the execution completes, the tool updates this marker to "Success" or "Failure". GitHub intercepts the merge execution request and keeps it locked until all specified mandatory statuses report green.

### Q6. What is a Code Owners file in GitHub, and how does it differ from standard requested reviews?

**Company asked:** *Oracle, Deloitte*

**Answer:**

A `CODEOWNERS` file automatically assigns specified individuals or engineering groups as mandatory reviewers when modifications alter files within designated directories. While standard PR reviews can be requested optionally from any collaborator, a `CODEOWNERS` protection rule can be set to block merges unless the designated owner explicitly approves the changes.

---

### Q7. How do you handle merge conflicts directly inside a GitHub Pull Request interface?

**Company asked:** *Mindtree*

**Answer:**

If a feature branch conflicts with updates made to the base branch, GitHub displays a conflict warning. If the conflicts are simple line mismatches, they can be resolved directly inside the browser using GitHub's web code editor. For complex or architectural structural changes, developers must pull the target branch locally, merge it into their feature branch, manually resolve the conflicts in an IDE, and push the updated branch back to GitHub.

### Q8. What is the difference between Git and GitHub?

**Company asked:** *TCS Walk-In Interview*

**Answer:**

* **Git:** A distributed open-source version control software tool run locally on a developer's machine to track local code file changes.
* **GitHub:** A cloud-hosted web platform that hosts Git repositories, providing a central platform for teamwork along with features like pull requests, issue tracking, and access management.

---

### Q9. Why should you avoid storing the Terraform state file (`terraform.tfstate`) in a Git repository?

**Company asked:** *Genpact*

**Answer:**

The Terraform state file contains clear-text data about your managed infrastructure, which often includes sensitive values like database passwords, API tokens, and initial keys. Storing this file in Git creates security risks and can cause synchronization issues when multiple developers make changes concurrently. Instead, the state file should be kept in a secure remote backend with state locking capabilities, such as an Azure Blob Storage container or an AWS S3 bucket.

---

### Q10. What role does a webhook play in connecting a code repository to an orchestration tool?

**Company asked:** *CGI*

**Answer:**

A webhook is an automated HTTP callback notification mechanism triggered by specific repository events, such as pushing code or opening a PR. When an event occurs, GitHub sends an HTTP POST request containing a payload to the external pipeline system's endpoint. This signal tells the automation engine to immediately pull the new changes and kick off the configured pipeline tasks.

## 4. DevOps Architect Tips & Intellegent Suggestions

> **Experience Perspective:** *As a DevOps Architect with over 20 years of experience designing secure, scalable software delivery pipelines, consider these three advanced recommendations for enterprise production environments:*

1. **Implement `CODEOWNERS` with Strict Monorepo Separation:**
In complex cloud setups, never rely solely on generic peer approvals. Use a structured `CODEOWNERS` layout to map specific infrastructure components to dedicated sub-teams. For example, any modifications inside a `/security-policies` folder should require an explicit sign-off from your security engineering lead, while modifications in network routing folders should require approval from the infrastructure operations group.
2. **Mandate Short-Lived Feature Branches to Prevent Merge Miseries:**
Avoid keeping long-running feature branches open for weeks, as they inevitably lead to complex merge conflicts. Enforce a development practice where feature branches are short-lived (ideally 2 to 3 days max) and focus on small, atomic changes. This team habit ensures code drifts remain minimal and makes automated code testing faster and more reliable.
3. **Embed Automated Cost and Policy Scans into the PR Review Interface:**
Do not limit your pull request status checks to basic syntax validation or security scanning. Introduce automated governance tools directly into the PR phase—such as **Infracost** for estimating changes in cloud spend, or **Open Policy Agent (OPA) / Rego** for checking compliance. Getting cost projections and corporate compliance scorecards directly inside the PR conversation allows reviewers to evaluate financial and architectural impacts before infrastructure gets deployed.
