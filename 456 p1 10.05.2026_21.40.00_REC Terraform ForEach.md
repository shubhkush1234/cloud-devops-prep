**3. Detailed Notes**

1. **Variables Classification**:
1. The session begins with a flowchart on draw.io breaking down variable types.
2. **Singular Types**: These hold a single value. Examples include **String** (Alphanumeric), **Number** (Numeric), and **Bool** (true/false).
3. **Plural Types**: These hold multiple values. Examples include **List**, **Set**, and **Map**.


2. **Exploring Plural Types in Detail**:
1. **List**: Contains the same type of elements and **can hold duplicate values**. Syntax shown: `["laal-lipistic", "laal-lipistic", "blue-lipistic"]`.
2. **Set**: Contains the same type of elements with **no duplicacy**. If you put duplicates in a set, Terraform filters them out.
3. **Map**: Works on key-value pairs. The trainer uses the example:
`{ lipstick = laal, blush = pink, powder = ponds, mascara = grey }`


3. **Looping in Terraform (Count vs. ForEach)**:
1. Loops are used to create multiple instances of a resource without duplicating code blocks.
2. **Count Loop**: The trainer writes "**useless- iska koi use nahi karta duniya me..**" under the Count loop. The reason is that `count` relies on list indices (0, 1, 2). If you delete index 0, the remaining elements shift up, causing Terraform to destroy and recreate the remaining resources.
3. **ForEach Loop**: The preferred method. The trainer notes that using `for_each` means "1 resource block = 1 real infra object in the cloud." It maps directly to the item's value or key, making it safe from index-shifting issues.


4. **Writing the Terraform Code**:
1. The trainer opens VS Code to write a configuration for Azure Resource Groups.
2. **Initial Code Attempt**:
```hcl
resource "azurerm_resource_group" "rg" {
  for_each = ["rg-lipistic", "rg-maskara", "rg-bindi"]
  name     = "rg-speaker"
  location = "West Europe"
}

```


3. The trainer runs `terraform plan` in the terminal.


5. **Encountering and Fixing the Tuple Error**:
1. The terminal throws a red error: **Error: Invalid for_each argument**.
2. The error message explicitly states: "The given "for_each" argument value is unsuitable: the "for_each" argument must be a map, or set of strings, and you have provided a value of type tuple."
3. **The Fix**: The `for_each` meta-argument does not accept simple lists (tuples). The trainer fixes this by utilizing the **toset()** function to convert the list into a set of strings.
4. **Corrected Code from Video**:
```hcl
resource "azurerm_resource_group" "rg" {
  for_each = toset(["rg-lipistic", "rg-maskara", "rg-bindi"])
  name     = each.value
  location = "West Europe"
}

```




6. **The Tablet Stripping Machine Analogy**:
1. To visually explain how `toset()` works, the trainer shows an image of a "Tablet Stripping Machine".
2. **Input**: A solid sheet of tablets (representing the List/Tuple).
3. **Machine**: The machine process (representing the `toset()` function).
4. **Output**: Individual, separated tablets (representing the Set of individual strings passed to the `for_each` loop).



**4. Interview Questions Section**

**5. Questions Discussed by Trainer in Video**

1. **What is the primary difference between a List and a Set in Terraform?**
1. **Answer**: A **List** is an ordered collection that allows duplicate values. A **Set** is an unordered collection that strictly prohibits duplicate values. If duplicate items are provided to a Set, Terraform automatically removes the duplicates.
2. 2. **Why does the `for_each` argument fail when provided with a direct list `["a", "b"]`?**


3. **Answer**: The **for_each** meta-argument is strictly designed to accept only a **Map** or a **Set of strings**. When a standard list is passed, Terraform interprets it as a tuple. You must explicitly convert the list to a set using the **toset()** function.
4. 3. **Why is the `count` loop considered dangerous or "useless" for dynamic resource creation?**


5. **Answer**: The **count** loop relies on numerical integer indexing (0, 1, 2). If an item is removed from the middle of the list, all subsequent items shift their index. Terraform interprets this shift as a massive configuration change, resulting in the destruction and recreation of resources, which can cause catastrophic downtime in production.



**6. Internet Researched Questions (Asked in Top Companies)**

1. **When would you choose `for_each` over `count`? (Asked in Amazon, Accenture)**
1. **Answer**: You should choose **for_each** over **count** when you need to create multiple resources that are not completely identical, or when the collection of resources might change over time. **for_each** uses keys instead of integer indices, so adding or removing an item from the middle of the collection won't trigger the recreation of all subsequent resources.
2. 2. **What happens to the Terraform state file when you switch a resource from `count` to `for_each`? (Asked in Microsoft, Infosys)**


3. **Answer**: Terraform tracks `count` resources using integer indices (e.g., `aws_instance.web[0]`) and `for_each` resources using string keys (e.g., `aws_instance.web["dev"]`). If you switch from `count` to `for_each` directly, Terraform will see the indexed resources as "deleted" and the keyed resources as "new," attempting to destroy and recreate everything. You must use the `terraform state mv` command to manually migrate the state addresses before applying.
4. 3. **Can you use `count` and `for_each` in the same resource block? (Asked in TCS, Wipro)**


5. **Answer**: No. Terraform strictly prohibits using both **count** and **for_each** within the exact same resource or module block. You must choose one looping meta-argument based on your specific use case.
6. 4. **How do you conditionally create a resource using the `count` meta-argument? (Asked in IBM, Capgemini)**


7. **Answer**: Conditional resource creation is achieved using a ternary operator with the **count** argument. If the condition evaluates to true, `count` is set to 1 (creating the resource). If false, `count` is set to 0 (skipping creation).
8. **Code (source: internet)**:
```hcl
resource "aws_instance" "web" {
  count         = var.deploy_instance ? 1 : 0
  ami           = "ami-123456"
  instance_type = "t2.micro"
}

```


9. 5. **What is the `each.key` and `each.value` object in Terraform? (Asked in Oracle, Cognizant)**


10. **Answer**: When using **for_each**, Terraform exposes the `each` object inside the resource block. If iterating over a Map, `each.key` corresponds to the map's key, and `each.value` corresponds to the value. If iterating over a Set of strings, both `each.key` and `each.value` hold the exact same string value.
11. 6. **What is a Terraform Module and how does `for_each` apply to it? (Asked in JPMorgan Chase, LTI)**


12. **Answer**: A **module** is a reusable container for a set of related resources. As of Terraform 0.13, you can use the **for_each** meta-argument directly on a `module` block to instantiate multiple copies of the entire module based on a map or set, passing unique variables to each instance.
13. 7. **Explain the concept of Terraform "Drift" and how loops impact it. (Asked in Google, Deloitte)**


14. **Answer**: **Drift** occurs when the real-world infrastructure is manually altered outside of Terraform, causing it to deviate from the `terraform.tfstate` file. When using loops, drift can be harder to spot if dynamic data sources are used to populate the loop. Running `terraform plan` detects this drift and attempts to revert the manual changes to match the code.
15. 8. **How do you generate dynamic nested blocks (like multiple security group rules) within a resource? (Asked in Mindtree, Cisco)**


16. **Answer**: While `for_each` creates entire resources, the **dynamic block** is used to create multiple nested configuration blocks *inside* a single resource. A dynamic block acts like a localized `for_each` loop specifically for nested arguments (like `ingress` or `egress` rules).
17. 9. **How would you iterate over a List of Objects using `for_each`? (Asked in Goldman Sachs, Barclays)**


18. **Answer**: Because `for_each` only accepts Maps or Sets, you cannot directly pass a List of Objects. You must use a `for` expression to transform the List of Objects into a Map, projecting a unique attribute from the object to act as the map's key.
19. **Code (source: internet)**:
```hcl
locals {
  server_map = { for server in var.server_list : server.name => server }
}

```


20. 10. **What is the difference between `variables.tf` and `terraform.tfvars`? (Asked in Red Hat, Infosys)**


21. **Answer**: **variables.tf** is where you *declare* the variables, specifying their type, description, and optional default values. **terraform.tfvars** is the file where you actually *assign* real data/values to those declared variables during deployment.



**7. Tips from a DevOps Architect (20+ Years Experience)**

1. **Never Hardcode Infrastructure Keys**: When using `for_each` with sets, avoid using dynamically generated data (like randomly generated IDs from another resource) as the set values. Terraform must know the keys *before* the apply phase (during the plan phase). Always rely on deterministic, known strings for your loop keys to maintain a stable state file.
2. **Master the `for` Expression for Data Munging**: As an architect, I rarely get perfect data structures from developers. You will frequently receive complex lists of JSON objects. Mastering Terraform's `for` loop to reconstruct Lists into Maps gives you the ultimate power to feed clean data into your `for_each` resource blocks seamlessly.
3. **Use `prevent_destroy` on Looped Stateful Resources**: When wrapping databases, storage accounts, or key vaults in a `for_each` loop, a simple typo in your map key can cause Terraform to destroy the resource and recreate it with the new key. Always add a `lifecycle { prevent_destroy = true }` block to stateful loop resources to safeguard against catastrophic data loss caused by accidental key renaming.
